###  Intuition
- Robots move either **left (L)** or **right (R)**.
- A collision happens only when:
  - A robot moving right meets a robot moving left.
- We simulate collisions using a **stack**:
  - Store indices of robots moving right.
  - When a left-moving robot appears, resolve collisions with stack top.

---

###  Approach (Step-by-Step)
1. **Combine and sort robots**
   - Store `(position, health, direction, original_index)`
   - Sort by position to simulate movement correctly.

2. **Use a stack**
   - Push indices of robots moving right (`R`).

3. **Collision handling**
   - When encountering a left-moving robot (`L`):
     - Compare with top of stack (right-moving robot).
     - Apply rules:
       - Smaller health → robot dies
       - Larger health → survives but loses 1 health
       - Equal → both die

4. **Update health dynamically**
   - Reduce health during collisions.
   - Mark dead robots.

5. **Collect survivors**
   - Return remaining robots’ health in **original order**.

---

###  Complexity
- **Time Complexity:** `O(n log n)`
  - Sorting dominates.
  - Each robot enters/exits stack at most once → `O(n)`
- **Space Complexity:** `O(n)`
  - For stack and tracking arrays.

---

###  Code Explanation (Stepwise)
1. `zip + sort`
   - Aligns all robot data and sorts by position.

2. `stack`
   - Tracks right-moving robots waiting for collision.

3. `while stack and current health > 0`
   - Resolves chain collisions.

4. Health updates:
   - Use tuple reassignment to update health after collision.

5. `alive[]`
   - Tracks whether a robot survives.

6. Final result:
   - Extract surviving robots.
   - Sort by original index.
   - Return only health values.
